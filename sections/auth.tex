%  Authentifizierung & Autorisierung
% ============================================================

Die Sicherheit einer Webanwendung steht und fällt mit zwei grundlegenden
Fragen: \textit{Wer ist der Nutzer?} -- das ist \textbf{Authentifizierung}
-- und \textit{Was darf der Nutzer?} -- das ist \textbf{Autorisierung}.
Für LearnHub wurde eine Lösung benötigt, die beide Aspekte abdeckt und
gleichzeitig mit dem bestehenden LDAP-System der HTL Leonding kompatibel
ist.

Dieses Kapitel führt zunächst die notwendigen Grundbegriffe ein, erklärt
den Authentifizierungsablauf anhand von Diagrammen und vergleicht
abschließend drei Lösungen.

% ----------------------------------------------------------
\subsection{Grundbegriffe}
% ----------------------------------------------------------

\subsubsection{Single Sign-On}

Single Sign-On (SSO) bezeichnet ein Verfahren, bei dem sich
Nutzer:innen einmalig anmelden und danach automatisch Zugriff auf
mehrere Anwendungen erhalten, ohne sich erneut authentifizieren zu
müssen.~\cite{ibm-sso} Für eine Schulplattform wie LearnHub ist das
besonders wertvoll: Schüler:innen und Lehrpersonen verwenden ihre
HTL-Zugangsdaten, die sie ohnehin täglich für andere Schulsysteme
nutzen -- ohne eine separate Registrierung oder ein weiteres Passwort.

\subsubsection{OAuth 2.0}

OAuth~2.0 ist ein offenes Autorisierungs-Framework, das es Anwendungen
erlaubt, im Namen eines Nutzers auf geschützte Ressourcen zuzugreifen,
ohne dass der Nutzer sein Passwort direkt an die Anwendung
weitergibt.~\cite{oauth2-rfc} Stattdessen erhält die Anwendung nach
erfolgreicher Anmeldung ein kurzlebiges \textit{Access Token}, das für
weitere Anfragen verwendet wird.

OAuth~2.0 definiert verschiedene \textit{Grant Types}, also
Ablaufmuster, je nach Anwendungsfall. Für browserbasierte Webanwendungen
wie LearnHub ist der \textbf{Authorization Code Flow mit PKCE}
(\textit{Proof Key for Code Exchange}) der empfohlene
Standard.~\cite{oauth2-rfc} Abbildung~\ref{fig:oauth-flow} zeigt diesen
Ablauf.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{./pics/oauth_flow.png}
    \caption{OAuth~2.0 Authorization Code Flow mit PKCE}
    \label{fig:oauth-flow}
\end{figure}

Das Diagramm zeigt den vollständigen Ablauf in fünf Schritten: Die
Anwendung leitet den Browser zunächst zur Loginseite des
Identity-Providers weiter und übergibt dabei einen zufällig generierten
\textit{Code Verifier} (PKCE). Nach erfolgreicher Anmeldung sendet der
Identity-Provider einen kurzlebigen \textit{Authorization Code} zurück
an die Anwendung. Diese tauscht den Code zusammen mit dem Code Verifier
gegen ein JWT ein. Ab diesem Zeitpunkt kann die Anwendung das JWT bei
jeder API-Anfrage als Nachweis der Identität mitsenden. PKCE stellt
dabei sicher, dass ein abgefangener Authorization Code ohne den
zugehörigen Code Verifier wertlos ist.~\cite{oauth2-rfc}

\subsubsection{OpenID Connect}

OpenID Connect (OIDC) ist eine Authentifizierungsschicht, die auf
OAuth~2.0 aufbaut.~\cite{openid-connect} Während OAuth~2.0 primär für
Autorisierung konzipiert ist, ermöglicht OpenID Connect die Verifikation
der Identität eines Nutzers. Dazu stellt OIDC zusätzlich zum Access
Token ein \textit{ID Token} bereit, das Informationen über den
angemeldeten Nutzer enthält. Keycloak implementiert OpenID Connect
vollständig.~\cite{openid-connect}

\subsubsection{LDAP}

LDAP (\textit{Lightweight Directory Access Protocol}) ist ein
standardisiertes Protokoll für den Zugriff auf
Verzeichnisdienste.~\cite{ldap-rfc} Schulen und Unternehmen verwenden
LDAP, um Benutzerkonten zentral in einer hierarchischen Baumstruktur
zu verwalten. Die HTL Leonding betreibt ein solches LDAP-Verzeichnis
für alle Schüler:innen und Lehrpersonen. Identity-Provider wie Keycloak
können direkt mit LDAP verbunden werden und die dort gespeicherten
Konten für die Anmeldung nutzen, ohne sie zu
duplizieren.~\cite{ldap-rfc}

\subsubsection{JSON Web Token}

Ein JSON Web Token (JWT) ist ein kompaktes, URL-sicheres Format zur
sicheren Übertragung von Informationen zwischen zwei
Parteien.~\cite{jwt-rfc} Nach erfolgreicher Anmeldung stellt der
Identity-Provider dem Frontend ein JWT aus, das bei jeder weiteren
Anfrage ans Backend mitgeschickt und dort validiert wird -- ohne dass
das Backend erneut den Identity-Provider kontaktieren
muss.~\cite{jwt-best-practices}

Ein JWT besteht aus drei durch Punkte getrennten, Base64url-kodierten
Teilen:~\cite{jwt-rfc} dem \textbf{Header} mit Token-Typ und
Signaturalgorithmus, dem \textbf{Payload} mit den eigentlichen Nutzdaten
(\textit{Claims}) sowie der \textbf{Signatur} als kryptografischem
Nachweis der Echtheit.

\paragraph{Signaturalgorithmus RS256}

LearnHub verwendet den Algorithmus \textbf{RS256}
(\textit{RSA Signature with SHA-256}).~\cite{jwt-rfc} Das Kürzel setzt
sich zusammen aus \textbf{RSA} (\textit{Rivest--Shamir--Adleman}), einem
asymmetrischen Kryptografieverfahren, und \textbf{SHA-256}
(\textit{Secure Hash Algorithm 256-bit}), einer kryptografischen
Hashfunktion.~\cite{jwt-rfc}

Im Gegensatz zu symmetrischen Verfahren wie HS256 -- bei denen
\textit{derselbe} geheime Schlüssel sowohl zum Signieren als auch zum
Verifizieren verwendet wird -- arbeitet RS256 mit einem
\textbf{Schlüsselpaar} aus zwei mathematisch verknüpften Schlüsseln:

\begin{itemize}
    \item \textbf{Privater Schlüssel (\textit{Private Key}):} Liegt
    ausschließlich beim Identity-Provider (Keycloak). Wird zum
    \textit{Signieren} des Tokens verwendet. Niemals weitergegeben.
    \item \textbf{Öffentlicher Schlüssel (\textit{Public Key}):} Wird
    vom Identity-Provider über einen öffentlich abrufbaren
    Discovery-Endpunkt (\texttt{/realms/\{realm\}/protocol/openid-connect/certs})
    bereitgestellt. Wird vom Backend zum \textit{Verifizieren} der
    Signatur verwendet.
\end{itemize}

Der Signiervorgang läuft wie folgt ab: Keycloak erzeugt aus dem
Base64url-kodierten Header und Payload einen SHA-256-Hash. Dieser Hash
wird anschließend mit dem privaten RSA-Schlüssel verschlüsselt -- das
Ergebnis ist die Signatur, der dritte Teil des JWT.~\cite{jwt-rfc}

Das Backend verifiziert die Signatur, indem es den öffentlichen Schlüssel
von Keycloak lädt, die Signatur damit entschlüsselt und den resultierenden
Hash mit einem selbst berechneten Hash aus Header und Payload vergleicht.
Stimmen beide überein, ist das Token authentisch und wurde nicht
manipuliert.~\cite{jwt-best-practices}

Der entscheidende Vorteil gegenüber HS256 liegt darin, dass das Backend
den privaten Schlüssel niemals kennen muss. Mehrere Dienste können
denselben öffentlichen Schlüssel zur Verifikation verwenden, ohne dass
ein kompromittiertes Backend automatisch zur Ausstellung gefälschter
Tokens missbraucht werden könnte.~\cite{jwt-best-practices}

Listing~\ref{lst:jwt-header} zeigt den Header eines realen JWT aus dem
HTL-System:

\begin{lstlisting}[language=json, caption={JWT Header -- RS256 Algorithmus mit Key ID},
                   label={lst:jwt-header}]
{
  "alg": "RS256",
  "typ": "JWT",
  "kid": "brdHR1646TJG1z2iSlElrRXwZMG8khgICgQKRFQd0sY"
}
\end{lstlisting}

Das Feld \texttt{kid} (\textit{Key ID}) identifiziert den verwendeten
Schlüssel eindeutig. Betreibt Keycloak mehrere Schlüsselpaare -- etwa
nach einer Schlüsselrotation -- kann das Backend anhand der Key ID
exakt den richtigen öffentlichen Schlüssel laden.~\cite{jwt-rfc}

\paragraph{JWT Payload}

Der Payload enthält die Informationen, die das Backend für
Authentifizierung und Autorisierung benötigt:

\begin{lstlisting}[language=json, caption={JWT Payload aus dem HTL Keycloak-Realm},
                   label={lst:jwt-payload}]
{
  "sub":                "cbd71d68-661c-4fd7-bced-3444f968d59d",
  "preferred_username": "it210999",
  "given_name":         "Max",
  "family_name":        "Mustermann",
  "email":
    "m.mustermann@students.htl-leonding.ac.at",
  "distinguishedName":
    "CN=it210999,OU=Students,OU=5BHITM,DC=htl-leonding,DC=local",
  "realm_access": {
    "roles": ["default-roles-2526_5bhitm", "offline_access"]
  },
  "exp": 1767552049,
  "iat": 1767551749
}
\end{lstlisting}

Das Feld \texttt{sub} enthält die eindeutige Keycloak-UUID des Nutzers,
die als stabiler Identifikator in der eigenen Datenbank verwendet wird.
\texttt{preferred\_username} und \texttt{email} liefern den
HTL-Benutzernamen und die E-Mail-Adresse direkt aus dem LDAP. Das Feld
\texttt{distinguishedName} ist besonders relevant: Es enthält den
vollständigen LDAP Distinguished Name und ist das einzige zuverlässige
Merkmal, um Schüler:innen von Lehrpersonen zu unterscheiden --
Schüler:innen haben \texttt{OU=Students} im DN, bei Lehrpersonen fehlt
dieser Eintrag. Die Felder \texttt{exp} und \texttt{iat} geben Ablauf-
und Ausstellungszeit als Unix-Timestamp an.~\cite{jwt-rfc}

\subsubsection{Rollenbasierte Zugriffskontrolle}

Rollenbasierte Zugriffskontrolle (\textit{Role-Based Access Control},
RBAC) ist ein Sicherheitsmodell, bei dem Berechtigungen nicht direkt
an einzelne Nutzer:innen, sondern an Rollen vergeben werden.~\cite{quarkus-security-jwt}
In LearnHub erfolgt die Rollenzuweisung vollautomatisch anhand des
\texttt{distinguishedName} im JWT -- eine manuelle Konfiguration ist
nicht notwendig.

\subsection{Authentifizierungsablauf in LearnHub}
% ----------------------------------------------------------

Abbildung~\ref{fig:auth-sequence} zeigt den vollständigen
Authentifizierungsablauf als Sequenzdiagramm. Es beschreibt den
Normalfall: eine Nutzerin bzw. ein Nutzer öffnet LearnHub und ist noch
nicht angemeldet.

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{./pics/auth_sequence.png}
    \caption{Sequenzdiagramm: Authentifizierungsablauf in LearnHub}
    \label{fig:auth-sequence}
\end{figure}

Der Ablauf beginnt damit, dass der \texttt{AuthGuard} beim Aufrufen
einer Route prüft, ob eine aktive Keycloak-Session existiert. Da keine
vorhanden ist, wird der Browser automatisch zur Keycloak-Loginseite der
HTL weitergeleitet. Die eingegebenen Zugangsdaten werden von Keycloak
gegen das LDAP-Verzeichnis validiert. Nach erfolgreicher Anmeldung
stellt Keycloak ein JWT aus und leitet zurück zur Anwendung. Das
Frontend ruft daraufhin den Backend-Endpunkt
\texttt{/api/users/register} auf, der den Nutzer in der eigenen
Datenbank anlegt oder -- bei wiederholtem Login -- nur zurückgibt. Ab
diesem Zeitpunkt fügt der \texttt{AuthInterceptor} das JWT automatisch
an jede ausgehende HTTP-Anfrage an, und das Backend validiert die
Signatur bei jedem Aufruf.~\cite{jwt-rfc, oauth2-rfc}

% ----------------------------------------------------------
\subsection{HTTP-Statuscodes der Authentifizierung}
% ----------------------------------------------------------

Für die Kommunikation zwischen Frontend und Backend verwendet LearnHub
standardisierte HTTP-Statuscodes, die klar kommunizieren, warum eine
Anfrage erfolgreich war oder scheiterte.~\cite{quarkus-security-jwt}
Tabelle~\ref{tab:http-status} zeigt die relevanten Codes im Kontext
der Authentifizierung:

\begin{table}[H]
    \centering
    \caption{HTTP-Statuscodes bei der Authentifizierung}
    \label{tab:http-status}
    \begin{tabular}{cll}
        \hline
        \textbf{Code} & \textbf{Bedeutung} & \textbf{Wann in LearnHub} \\
        \hline
        200 & OK & Anfrage erfolgreich, Nutzer bereits vorhanden \\
        201 & Created & Neuer Nutzer wurde beim ersten Login angelegt \\
        401 & Unauthorized & Kein Token oder Token ungültig/abgelaufen \\
        403 & Forbidden & Token gültig, aber Rolle reicht nicht aus \\
        404 & Not Found & Ressource existiert nicht \\
        \hline
    \end{tabular}
\end{table}

Der Unterschied zwischen \textbf{401} und \textbf{403} ist dabei
besonders relevant: HTTP~401 bedeutet, dass die Identität der
Nutzerin bzw. des Nutzers nicht festgestellt werden konnte -- entweder
fehlt das Token vollständig oder es ist nicht mehr gültig. HTTP~403
hingegen bedeutet, dass die Person zwar bekannt und angemeldet ist,
aber nicht die erforderliche Berechtigung für diese Aktion besitzt.
Im Frontend führt ein 401-Fehler zu einer automatischen Weiterleitung
zur Keycloak-Loginseite, während ein 403-Fehler eine entsprechende
Fehlermeldung anzeigt.

\subsection{Vergleich der Authentifizierungslösungen}
% ----------------------------------------------------------

Für LearnHub wurden drei verbreitete Lösungen evaluiert.

\subsubsection{Keycloak}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.55\textwidth]{./pics/keycloak_login.png}
    \caption{Keycloak-Loginseite der HTL Leonding}
    \label{fig:keycloak-login}
\end{figure}

Keycloak ist eine Open-Source-Lösung für Identity and Access Management
(IAM), entwickelt von Red Hat.~\cite{keycloak-docs} Es unterstützt
OAuth~2.0, OpenID Connect und SAML~2.0 und bietet SSO, User Federation
über LDAP sowie Multi-Faktor-Authentifizierung. Abbildung
\ref{fig:keycloak-login} zeigt die Loginseite der HTL Leonding: Sie
ist direkt in den schulischen Kontext eingebettet und erfordert keine
separate Registrierung.

Ein zentrales Architekturprinzip von Keycloak ist die Trennung der
Authentifizierung von der Anwendungslogik.~\cite{keycloak-docs} Die
Anwendung selbst speichert zu keinem Zeitpunkt Passwörter -- das
übernimmt ausschließlich Keycloak. Für das Team bedeutete das eine
klare Aufgabenverteilung: Backend und Frontend kümmern sich nur um die
Anwendungslogik, Keycloak kümmert sich um die Sicherheit.

\paragraph{Integration mit dem HTL-Realm}

Für LearnHub stellte die HTL Leonding einen bereits vorkonfigurierten
Keycloak-Realm (\texttt{2526\_5bhitm}) zur Verfügung, der direkt mit
dem LDAP-Verzeichnis verbunden ist. Das Team erhielt Zugriff auf den
Client \texttt{frontend} -- eine eigene Keycloak-Instanz musste nicht
aufgesetzt werden.

Das bedeutet in der Praxis: Jede Person mit einem HTL-LDAP-Account --
also jede Schülerin, jeder Schüler, jede Lehrperson -- kann sich sofort
bei LearnHub anmelden. Neue Jahrgänge werden automatisch verfügbar,
sobald die Schule sie im LDAP anlegt. Ausgeschiedene Nutzer:innen
verlieren den Zugriff automatisch mit der Deaktivierung des
LDAP-Accounts. Die Plattform nutzt damit genau die Infrastruktur, die
an der Schule ohnehin vorhanden ist, und vermeidet jeden zusätzlichen
Verwaltungsaufwand.~\cite{keycloak-docs}

\textbf{Vorteile:}
\begin{itemize}
    \item Native LDAP-Integration: alle HTL-Accounts sofort verfügbar,
    keine separate Benutzerverwaltung notwendig
    \item Vollständige Datenkontrolle durch on-premise Betrieb
    \item HTL-Realm bereits vorkonfiguriert und einsatzbereit
    \item Keine laufenden Lizenzkosten
\end{itemize}

\textbf{Nachteile:}
\begin{itemize}
    \item Kein direkter Zugriff auf die Realm-Konfiguration
    \item Eigene Testkonten konnten nicht angelegt werden
\end{itemize}

\subsubsection{Auth0}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.55\textwidth]{./pics/auth0_login.png}
    \caption{Typische Auth0 Universal Login-Seite~\cite{auth0-docs}}
    \label{fig:auth0-login}
\end{figure}

Auth0 ist eine cloud-basierte Identity-as-a-Service-Lösung (IDaaS) von
Okta.~\cite{auth0-docs} Das Kernkonzept von Auth0 ist, die gesamte
Authentifizierungsinfrastruktur als vollständig verwalteten Cloud-Service
bereitzustellen: Entwickler:innen müssen keine Server betreiben, keine
Updates einspielen und sich nicht um Skalierung kümmern.~\cite{auth0-architecture}

Ein Hauptmerkmal ist das sogenannte \textit{Universal Login}: Auth0
stellt eine vorgefertigte, anpassbare Loginseite bereit (siehe
Abbildung~\ref{fig:auth0-login}), die Entwickler:innen mit wenigen
Zeilen Code in jede Anwendung einbinden können.~\cite{auth0-docs} Die
Loginseite kann mit eigenem Branding versehen werden und unterstützt
direkt Social Logins über Google, GitHub, Microsoft und viele weitere
Anbieter -- ohne eigene Integration. Für die Entwicklung stellt Auth0
SDKs für JavaScript, Angular, React, Vue.js, Node.js, Java und viele
weitere Sprachen und Frameworks bereit.~\cite{auth0-docs}

Für Webanwendungen verwendet Auth0 ebenfalls den Authorization Code
Flow mit PKCE, der in Abschnitt~\ref{fig:oauth-flow} beschrieben
wurde. Der wesentliche Unterschied zu Keycloak liegt darin, dass die
gesamte Infrastruktur -- Token-Ausstellung, Schlüsselverwaltung,
Benutzerdaten -- in Auth0s Cloud liegt und nicht auf eigenen
Servern.~\cite{auth0-architecture}

Für LearnHub scheidet Auth0 aus zwei Gründen aus: Die LDAP-Anbindung
an das HTL-Verzeichnis ist ausschließlich in kostenpflichtigen
Enterprise-Tarifen verfügbar.~\cite{auth0-pricing} Ohne diese
Integration müssten alle HTL-Accounts manuell in Auth0 angelegt werden
-- was den zentralen Vorteil des schulischen Verzeichnisses zunichte
machen würde. Hinzu kommt, dass alle Authentifizierungsdaten bei einem
externen Anbieter in der Cloud liegen würden.

\textbf{Vorteile:}
\begin{itemize}
    \item Sehr schnelle Integration durch SDKs und Universal Login
    \item Kein eigener Server notwendig
    \item Umfangreiche Dokumentation und großer Funktionsumfang
    \item Social Login direkt integriert
\end{itemize}

\textbf{Nachteile:}
\begin{itemize}
    \item LDAP-Integration nur in teuren Enterprise-Tarifen~\cite{auth0-pricing}
    \item Authentifizierungsdaten liegen bei einem externen Anbieter
    \item Abhängigkeit von einem kommerziellen Cloud-Service
\end{itemize}

\subsubsection{Google OAuth 2.0}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.45\textwidth]{./pics/google_oauth_login.png}
    \caption{Typische Google OAuth 2.0 Anmeldeseite~\cite{google-oauth}}
    \label{fig:google-oauth-login}
\end{figure}

Google OAuth~2.0 ist einer der verbreitetsten Social-Login-Mechanismen
und ermöglicht es Nutzer:innen, sich mit ihrem bestehenden Google-Konto
bei Drittanwendungen anzumelden.~\cite{google-oauth} Das bekannte
Anmeldefenster (Abbildung~\ref{fig:google-oauth-login}) ist vielen
Nutzer:innen von zahlreichen Webseiten vertraut. Der Ablauf folgt dem
OAuth~2.0 Authorization Code Flow: Google fungiert dabei als
Identity-Provider, stellt nach der Anmeldung ein JWT aus und liefert
über OpenID Connect die Nutzerinformationen an die
Anwendung.~\cite{google-oauth}

\paragraph{Verwendung und Integration}

Für Entwickler:innen ist Google OAuth über die Google Cloud Console
einzurichten: Dort wird ein OAuth-Client angelegt, der eine
\textit{Client ID} und ein \textit{Client Secret} erhält. Die
Webanwendung leitet Nutzer:innen zur Google-Loginseite weiter und
empfängt nach erfolgreicher Anmeldung einen Authorization Code, den sie
gegen ein Access Token und ID Token eintauscht.~\cite{google-oauth}

Google stellt für die clientseitige Integration die
\textit{Google Identity Services Library} bereit, die mit wenigen
Zeilen JavaScript eingebunden werden kann:~\cite{google-oauth}

\begin{lstlisting}[style=htmlstyle,
                   caption={Einbindung von Google Sign-In im Frontend},
                   label={lst:google-signin}]
<script src="https://accounts.google.com/gsi/client"></script>
<div id="g_id_onload"
     data-client_id="YOUR_CLIENT_ID"
     data-callback="handleCredentialResponse">
</div>
<div class="g_id_signin" data-type="standard"></div>
\end{lstlisting}

Für Angular-Projekte gibt es zusätzlich die Bibliothek
\texttt{@abacritt/angularx-social-login}, die Google Sign-In als
Angular-Service kapselt.~\cite{google-oauth} Social Connections zu
anderen Anbietern wie GitHub, Microsoft oder Apple lassen sich nach
demselben Prinzip hinzufügen.

Das erhaltene ID Token ist ebenfalls ein JWT, das Informationen wie
Name, E-Mail-Adresse und die eindeutige Google-ID des Nutzers
enthält.~\cite{google-oauth} Es kann im Backend mit dem öffentlichen
Google-Schlüssel verifiziert werden -- analog zum RS256-Verfahren in
LearnHub.

\paragraph{Warum Google OAuth für LearnHub nicht geeignet ist}

Google OAuth ist für öffentliche Anwendungen, bei denen Nutzer:innen
ohnehin Google-Konten besitzen, eine hervorragende Lösung. Für LearnHub
ergibt sich jedoch ein grundsätzliches Problem: Es gibt keine
Möglichkeit, die Anmeldung auf das HTL-LDAP zu beschränken. Beliebige
Google-Konten könnten sich anmelden -- die automatische Unterscheidung
zwischen Schüler:innen und Lehrpersonen über den
\texttt{distinguishedName} wäre nicht möglich, da dieser nur im
HTL-LDAP gespeichert ist.~\cite{google-oauth} Die schulische
Infrastruktur würde damit vollständig umgangen.

\textbf{Vorteile:}
\begin{itemize}
    \item Sehr einfache Integration, Nutzer:innen kennen das
    Anmeldefenster bereits
    \item Keine eigene Passwortverwaltung notwendig
    \item Social Connections zu weiteren Anbietern direkt integrierbar
    \item Umfangreiche SDKs für alle gängigen Frameworks
\end{itemize}

\textbf{Nachteile:}
\begin{itemize}
    \item Keine Integration mit dem HTL-LDAP möglich
    \item Beliebige Google-Konten könnten sich anmelden
    \item Keine automatische Rollenunterscheidung (Schüler:in/Lehrperson)
    \item Abhängigkeit von Googles Infrastruktur
\end{itemize}

\subsubsection{Entscheidung: Keycloak}

Keycloak ist die einzige der drei evaluierten Lösungen, die alle
Anforderungen erfüllt. Die native LDAP-Integration ist der
entscheidende Faktor: Sie bedeutet, dass LearnHub die an der HTL
bereits vorhandene Infrastruktur vollständig nutzt. Wer an der Schule
einen Account hat, hat automatisch Zugang -- und wer keinen hat, auch
keinen Zugang. Dieses Prinzip lässt sich mit Auth0 nur gegen hohe
Kosten und mit Google OAuth gar nicht umsetzen. Hinzu kommen die
vollständige Kostenfreiheit, die Datenkontrolle auf schuleigenen
Servern und der bereits vorkonfigurierte HTL-Realm, der den
Entwicklungsstart wesentlich beschleunigte.~\cite{keycloak-docs}
