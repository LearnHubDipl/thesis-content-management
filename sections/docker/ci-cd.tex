\label{sec:cicd}

Aktuell läuft das Deployment von LearnHub manuell. Bei jeder 
Code-Änderung werden die Docker-Images von Hand gebaut, auf Docker 
Hub hochgeladen und auf der VM neu gestartet. Das funktioniert für 
den aktuellen Projektumfang, ist aber auf Dauer fehleranfällig und 
zeitaufwendig. Eine CI/CD-Pipeline würde diesen Prozess vollständig 
automatisieren.

CI steht für \textbf{Continuous Integration}: Code-Änderungen werden 
laufend zusammengeführt und automatische Tests prüfen dabei die 
Qualität. CD bedeutet \textbf{Continuous Deployment}: Getesteter 
Code wird automatisch in die Produktion gebracht. Zusammen bilden 
beide eine durchgehende Automatisierung vom ersten Commit bis zur 
laufenden Anwendung \cite{fowler2006continuous}.

\subsubsection{GitHub Actions}

Für LearnHub bietet sich \textbf{GitHub Actions} an, da der Code 
bereits auf GitHub liegt und GitHub Actions direkt integriert ist. 
Ein Workflow wird in einer YAML-Datei unter 
\texttt{.github/workflows/} definiert und von GitHub automatisch 
ausgeführt wenn ein bestimmtes Ereignis eintritt, zum Beispiel ein 
Push auf den Main-Branch \cite{github-actions}.

\subsubsection{Aufbau des Workflows}

Der Workflow besteht aus drei Jobs die nacheinander ablaufen:

\textbf{Job 1 und 2: Build und Test}

Die ersten zwei Jobs laufen parallel und sind für Backend und 
Frontend zuständig. Beim Backend wird der Code mit Maven gebaut 
und die Tests werden ausgeführt. Beim Frontend werden die 
Abhängigkeiten installiert, die Tests laufen und das Docker-Image 
wird gebaut. Beide Jobs taggen das fertige Image mit dem 
Commit-Hash, sodass später nachvollziehbar ist welches Image aus 
welchem Code entstanden ist. Anschließend werden die Images auf 
Docker Hub hochgeladen. Sensible Informationen wie Docker Hub 
Passwort oder SSH-Keys werden als \textbf{GitHub Secrets} 
gespeichert und im Workflow über 
\texttt{\$\{\{ secrets.NAME \}\}} referenziert, sodass sie 
nie im Quellcode stehen \cite{github-secrets}.

\textbf{Job 3: Deploy}

Der Deploy-Job startet erst wenn beide vorherigen Jobs erfolgreich 
waren. Das wird über die \texttt{needs} Direktive gesteuert. 
Zusätzlich läuft dieser Job nur bei einem Push auf den Main-Branch, 
nicht bei Pull Requests. Der Job baut eine SSH-Verbindung zur VM 
auf und führt dort zwei Befehle aus: \texttt{docker compose pull} 
lädt die neuen Images, \texttt{docker compose up -d} startet die 
betroffenen Container neu.

\subsubsection{Automatisierte Tests}

Ein wesentlicher Teil einer CI/CD-Pipeline sind automatisierte 
Tests. Für das Quarkus-Backend eignen sich Unit-Tests mit JUnit 5 
sowie Integration-Tests mit \texttt{@QuarkusTest}, womit die 
Anwendung in einer Testumgebung gestartet und HTTP-Requests 
simuliert werden können \cite{quarkus-testing}. Für die 
Datenbank-Tests kann eine In-Memory H2-Datenbank verwendet werden, 
um unabhängig von der produktiven MySQL-Datenbank zu bleiben.

Für das Angular-Frontend eignen sich Unit-Tests mit Jasmine und 
Karma für einzelne Components und Services, sowie End-to-End-Tests 
mit Cypress oder Playwright die reale Benutzerinteraktionen im 
Browser simulieren \cite{cypress-docs}.

\subsubsection{Rollback}

Da jedes Image mit einem Versions-Tag versehen wird, ist ein 
Rollback einfach: Im Compose File wird der Image-Tag auf eine 
frühere Version geändert und die Container werden neu gestartet. 
Die letzten drei bis fünf Versionen sollten dafür in der Registry 
aufbewahrt werden. Ein automatisierter Health-Check nach jedem 
Deployment kann fehlerhafte Deployments erkennen und die Pipeline 
kann bei Bedarf automatisch zur vorherigen Version zurückrollen.
