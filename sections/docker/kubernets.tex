\label{sec:kubernetes}

Kubernetes repräsentiert eine Plattform zur Orchestrierung von Containern und
findet primär Anwendung im Betrieb verteilter, skalierbarer Applikationen. Im
Unterschied zu Docker Compose, das hauptsächlich für das koordinierte Starten
mehrerer Container auf einem einzelnen Host konzipiert ist, fokussiert sich
Kubernetes auf den Betrieb von Containern in Cluster-Umgebungen mit multiplen
Knoten \cite{kubernetes-docs}.

\subsubsection{Grundidee von Kubernetes}

Kubernetes abstrahiert Container und Hosts zu logischen Konstrukten. Anwendungen
werden nicht mehr als isolierte Container betrachtet, sondern als sogenannte
\textit{Pods}, die einen oder mehrere eng gekoppelte Container umfassen können.
Mehrere Pods werden über \textit{Services} zugänglich gemacht, während
\textit{Deployments} und \textit{ReplicaSets} für die angestrebte Anzahl
operierender Instanzen verantwortlich sind.

Ein charakteristisches Merkmal von Kubernetes ist die deklarative Spezifikation
des Systemzustands. Der angestrebte Zustand (beispielsweise Anzahl der Replikate
oder verfügbare Services) wird definiert, während Kubernetes eigenständig dafür
sorgt, diesen Zustand zu realisieren und aufrechtzuerhalten.

\subsubsection{Typische Einsatzszenarien}

Kubernetes kommt insbesondere in folgenden Szenarien zum Einsatz:
\begin{itemize}
    \item hochskalierbare Webanwendungen mit stark fluktuierender Last,
    \item Microservice-Architekturen mit zahlreichen autonomen Services,
    \item Cloud-Umgebungen mit multiplen Hosts oder Rechenzentren,
    \item Systeme mit erhöhten Anforderungen an Verfügbarkeit und Ausfallsicherheit.
\end{itemize}

Durch Funktionalitäten wie automatisches Neustarten, Self-Healing und horizontale
Skalierung eignet sich Kubernetes besonders für umfangreiche und komplexe Systeme,
bei denen manuelle Interventionen minimiert werden sollen.

\subsubsection{Abgrenzung zu Docker Compose}

Im vorliegenden Projekt wird bewusst auf die Implementierung von Kubernetes
verzichtet. Der Funktionsumfang von Docker Compose deckt den gegebenen
Anwendungsfall vollständig ab. Die Anwendung besteht aus einer überschaubaren
Anzahl an Services und wird auf einer einzelnen virtuellen Maschine betrieben.

Während Kubernetes zusätzliche Komponenten wie eine Control Plane, Worker Nodes
und ein komplexes Netzwerkmodell benötigt, bleibt Docker Compose signifikant
einfacher in der Konfiguration und Wartung. Für kleinere Projekte oder
Bildungsprojekte überwiegt daher der Nutzen der reduzierten Komplexität gegenüber
den erweiterten Möglichkeiten von Kubernetes.

\subsubsection{Bewusste Technologieentscheidung}

Die Entscheidung gegen Kubernetes stellt keine technische Limitierung dar,
sondern resultiert aus einer bewussten Abwägung zwischen Nutzen und Aufwand. Der
Betrieb eines Kubernetes-Clusters hätte zusätzliche Konfigurations- und
Wartungsarbeiten erfordert, ohne einen direkten Mehrwert für den aktuellen
Projektumfang zu generieren.

Dennoch ist das Projekt so konzipiert, dass eine spätere Migration zu Kubernetes
grundsätzlich realisierbar wäre. Die eindeutige Separation der Services sowie die
vollständige Containerisierung bilden dafür eine geeignete Basis.

\subsubsection{Einordnung im Projektkontext}

Zusammenfassend lässt sich konstatieren, dass Kubernetes eine leistungsfähige
Plattform für großskalige, verteilte Systeme darstellt. Für den vorliegenden
Anwendungsfall ist jedoch der Einsatz von Docker Compose in Verbindung mit einer
virtuellen Maschine die zweckmäßigere Lösung. Dieser Ansatz realisiert einen
praxisnahen Produktivbetrieb bei gleichzeitig überschaubarem Administrationsaufwand.
