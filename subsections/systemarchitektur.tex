LearnHub folgt einem klassischen \textbf{Client-Server-Ansatz}. Das 
Frontend läuft im Browser und ist für die Darstellung der Oberfläche 
zuständig. Das Backend verarbeitet alle Anfragen, enthält die 
Geschäftslogik und kommuniziert mit der Datenbank. Die Kommunikation 
zwischen Frontend und Backend läuft über REST-APIs, der Datenaustausch 
erfolgt im JSON-Format. Durch diese klare Aufteilung wird das Prinzip 
der \textbf{Separation of Concerns} konsequent umgesetzt, also dass 
jede Schicht nur für ihren eigenen Bereich zuständig ist und nicht 
in die Aufgaben anderer eingreift.

\subsubsection{Frontend}

Das Frontend wurde mit Angular als Single-Page-Application umgesetzt. 
Das bedeutet dass Seitenwechsel ohne Neuladen der Seite stattfinden, 
was ein flüssiges Nutzererlebnis ermöglicht. Das Frontend besteht 
aus mehreren Angular-Components, die jeweils für einen bestimmten 
Bereich zuständig sind, etwa den Fragenkonfigurator, den Maturaupload 
oder den persönlichen Bereich.

Die Kommunikation mit dem Backend ist in Services gekapselt. Diese 
Services kümmern sich um die HTTP-Anfragen und liefern den Components 
die benötigten Daten. Durch diese Aufteilung zwischen Components und 
Services wird die Separation of Concerns auch innerhalb des Frontends 
umgesetzt: Components sind nur für die Darstellung zuständig, 
Services für die Logik und Datenbeschaffung. Das macht den Code 
leichter wartbar und einzelne Teile können unabhängig voneinander 
geändert oder wiederverwendet werden.

Für die Navigation verwendet das Frontend Angular-Routing. Abhängig 
von der Rolle des angemeldeten Nutzers werden unterschiedliche Routen 
freigeschaltet. Ein Schüler sieht andere Navigationspunkte und hat 
Zugriff auf andere Bereiche als eine Lehrkraft. Diese rollenbasierte 
Navigation wird direkt im Frontend über Route Guards gesteuert, wobei 
die eigentliche Berechtigungsprüfung immer im Backend stattfindet.

\subsubsection{Backend}

Das Backend wurde mit Java und Quarkus entwickelt und verarbeitet 
alle Anfragen aus dem Frontend. Auch hier wird die Separation of 
Concerns durch eine klare Schichtenarchitektur umgesetzt:

\begin{itemize}
    \item \textbf{Boundary}: REST-Endpunkte für die Kommunikation 
    mit dem Frontend. Diese Schicht nimmt Anfragen entgegen und 
    gibt Antworten zurück, enthält aber keine Geschäftslogik.
    
    \item \textbf{Model}: Datenentitäten die direkt auf die 
    Datenbanktabellen gemappt werden. Jede Entität repräsentiert 
    ein Objekt aus der Anwendungsdomäne, also zum Beispiel einen 
    User, ein Fach oder eine Frage.
    
    \item \textbf{Repository}: Zuständig für den Datenbankzugriff. 
    Die Boundary kommuniziert nicht direkt mit der Datenbank, 
    sondern immer über das Repository.
    
    \item \textbf{DTO (Data Transfer Object)}: Datenstrukturen 
    die speziell für den Transfer zwischen Frontend und Backend 
    designed sind. DTOs enthalten nur die Felder die das Frontend 
    tatsächlich braucht, nicht das gesamte Datenmodell.
    
    \item \textbf{Mapper}: Übersetzt zwischen Models und DTOs. 
    Dadurch ist das interne Datenmodell vom Frontend entkoppelt 
    und kann sich unabhängig davon weiterentwickeln.
    
    \item \textbf{Security}: Integration mit Keycloak für 
    Authentifizierung und Berechtigungschecks. Alle 
    Sicherheitsprüfungen laufen ausschließlich in dieser Schicht 
    ab, damit keine sicherheitsrelevante Logik im Frontend 
    umgesetzt wird.
\end{itemize}

\subsubsection{Authentifizierung und Autorisierung}

Die Anmeldung läuft über Keycloak, der von der HTL Leonding 
bereitgestellt wird und mit dem schulischen LDAP-Verzeichnis 
verbunden ist. Nach erfolgreicher Anmeldung erhält das Frontend 
ein JWT, das bei jeder Anfrage ans Backend mitgeschickt wird. 
Das Backend liest die Rolle aus dem Token und entscheidet damit 
welche Ressourcen der Nutzer erreichen darf. Schüler:innen haben 
andere Berechtigungen als Lehrer:innen, etwa beim Freigeben von 
Fragen oder dem Verwalten von Fächern und Themenpools. Da alle 
Sicherheitsprüfungen im Backend stattfinden, kann die 
Zugriffskontrolle nicht durch Manipulation des Frontends umgangen 
werden.

\subsubsection{Datenbank}

Alle relevanten Daten, also Benutzer, Fächer, Themenpools, Fragen 
und Mitschriften, werden in einer relationalen Datenbank gespeichert. 
Relationale Datenbanken eignen sich hier besonders gut, weil die 
Daten stark miteinander verknüpft sind. Eine Frage gehört zu einem 
Themenpool, ein Themenpool zu einem Fach, eine Mitschrift zu einem 
Themenpool und einem Benutzer. Diese Beziehungen lassen sich in 
einer relationalen Datenbank sauber über Fremdschlüssel abbilden.

Für die lokale Entwicklung wird H2 als In-Memory-Datenbank 
verwendet, da diese ohne Installation direkt in der Anwendung 
läuft und beim Start automatisch das Schema erstellt. Im 
Produktivbetrieb auf der VM kommt MySQL zum Einsatz, das in 
einem eigenen Docker-Container läuft und seine Daten in einem 
persistenten Volume speichert. Der Wechsel zwischen H2 und MySQL 
wird über das Quarkus-Profil gesteuert, also \texttt{dev} für 
die Entwicklung und \texttt{prod} für den Produktivbetrieb, 
ohne dass der Code selbst geändert werden muss.
