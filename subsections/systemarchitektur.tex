LearnHub folgt einem klassischen \textbf{Client-Server-Ansatz}. 
Das Frontend wurde mit Angular als Single-Page-Application 
umgesetzt, das Backend mit Java und Quarkus. Die Kommunikation 
zwischen beiden läuft über REST-APIs im JSON-Format, wobei das 
Prinzip der \textbf{Separation of Concerns} konsequent umgesetzt 
wird.

\subsubsection{Datenbank}

Alle relevanten Daten, also Benutzer, Fächer, Themenpools, Fragen 
und Mitschriften, werden in einer relationalen Datenbank gespeichert. 
Relationale Datenbanken eignen sich hier besonders gut, weil die 
Daten stark miteinander verknüpft sind. Eine Frage gehört zu einem 
Themenpool, ein Themenpool zu einem Fach, eine Mitschrift zu einem 
Themenpool und einem Benutzer. Diese Beziehungen lassen sich sauber 
über Fremdschlüssel abbilden.

Für die lokale Entwicklung wird H2 als In-Memory-Datenbank 
verwendet, die ohne Installation direkt in der Anwendung läuft 
und beim Start automatisch das Schema erstellt. Im Produktivbetrieb 
auf der VM kommt MySQL zum Einsatz, das in einem eigenen 
Docker-Container läuft und seine Daten in einem persistenten 
Volume speichert. Der Wechsel zwischen H2 und MySQL wird über 
das Quarkus-Profil gesteuert, also \texttt{dev} für die 
Entwicklung und \texttt{prod} für den Produktivbetrieb, 
ohne dass der Code selbst geändert werden muss.

    \caption{Datenmodell von LearnHub}
    \label{fig:datenmodell}
\end{figure}

Das Datenmodell spiegelt direkt die Datenbankstruktur von LearnHub 
wider. Jede Entität entspricht einer Tabelle in der MySQL-Datenbank, 
die Verbindungen zwischen den Entitäten werden als Fremdschlüssel 
abgebildet.

Der erste Bereich umfasst die Struktur der Plattform. 
\textbf{Subject} entspricht der Tabelle für Fächer, 
\textbf{TopicPool} der Tabelle für Themenpools, wobei jeder 
Themenpool über einen Fremdschlüssel einem Fach zugeordnet ist. 
\textbf{TopicContent} repräsentiert eine Mitschrift und speichert 
den Pfad zur PDF-Datei auf dem Server sowie den Freigabestatus 
(\texttt{isApproved}). Solange \texttt{isApproved} false ist, 
ist die Mitschrift nur für die Ersteller:in sichtbar. Erst wenn 
eine Lehrkraft die Mitschrift freigibt, wird der Wert auf true 
gesetzt und die Mitschrift ist öffentlich sichtbar. In der 
tatsächlichen Implementierung werden die Metadaten einer 
Mitschrift, also Freigabestatus, zuständige Lehrkraft und 
Zeitstempel, nicht in der Datenbank, sondern in einer 
gleichnamigen \texttt{meta.json} Datei direkt neben der PDF 
auf dem Server gespeichert. Dies ermöglicht eine einfache 
Aktualisierung des Status ohne Datenbankzugriff, hat jedoch 
den Nachteil dass keine strukturierten Datenbankabfragen 
über den Status möglich sind.

Der zweite Bereich umfasst das Fragensystem. \textbf{Question} 
speichert den Fragetext sowie den Typ (\texttt{QuestionType}), 
also ob es sich um eine Freitext- oder Multiple-Choice-Frage 
handelt. Bei Multiple-Choice-Fragen werden die Antwortmöglichkeiten 
in der \textbf{Answer} Tabelle gespeichert, wobei das Feld 
\texttt{isCorrect} angibt welche Antworten korrekt sind. Jede 
Frage ist über einen Fremdschlüssel einem Themenpool zugeordnet.

Sowohl \textbf{TopicContent} als auch \textbf{Question} können 
über die \textbf{MediaFile} Tabelle Bilder oder Grafiken verknüpft 
haben. Die \textbf{User} Tabelle ist mit beiden Hauptbereichen 
verknüpft, über Fremdschlüssel wird gespeichert welcher User 
welche Mitschrift hochgeladen und welche Frage erstellt hat. 
Das ermöglicht dass Schüler:innen nur ihre eigenen Mitschriften 
bearbeiten können, da das Backend bei jeder Anfrage prüft ob 
der anfragende User der Ersteller des Inhalts ist.
